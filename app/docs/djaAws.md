# SUMMARY
Probed the public msaexp-nirspec S3 bucket and DJA (dawn-cph.github.io) site directly with curl/WebFetch. Bucket-level ListBucket (XML listing) is denied for anonymous users (AccessDenied on ?list-type=2 and ?prefix=... for both path-style and virtual-hosted-style URLs, both s3.amazonaws.com and region endpoint) — so no directory enumeration is possible via S3 API. However individual object GetObject is public (HEAD/GET returns 200), so structure was reconstructed by downloading the emission-line catalog itself (which lists every file/root) rather than by listing. Confirmed the DJA nirspec_extractions/thumb/other APIs on grizli-cutout.herokuapp.com, verified spec.fits FITS structure with astropy, confirmed PNG previews exist, and confirmed neither S3 nor the Heroku API sets CORS headers on GET responses.

1) msaexp-nirspec bucket / extractions/ prefix: Bucket does not allow anonymous ListBucket (curl -s 'https://s3.amazonaws.com/msaexp-nirspec?list-type=2&prefix=extractions/&delimiter=/' -> AccessDenied, tried path-style, virtual-hosted, and us-east-1 regional endpoint, all denied). Individual objects ARE publicly gettable. Reconstructed structure from the catalog CSV itself: under extractions/ there is one subdirectory per "root" (mask/pointing) plus flat catalog/summary files at the extractions/ top level (dja_msaexp_emission_lines_v4.4.csv.gz, v4.5.csv.gz, nirspec_public_v4.4.html, nirspec_public_v4.5.html, nirspec_graded.html, nirspec_graded_v3.html, and per-grating concatenated spectra FITS files named dja_msaexp_emission_lines_v4.4.<grating-filter>_spectra.fits per the v4 blog post, e.g. .prism_spectra.fits, .g395m-f290lp_spectra.fits etc.). Counted 219 distinct root directories present in the v4.5 catalog (dja_msaexp_emission_lines_v4.5.csv.gz), all with names ending in "-v4" (e.g. a370-snobs4-pierel-v4, abell2744-castellano1-v4, capers-cos01-v4, rubies-egs62-v4). File naming pattern for spec files: extractions/<root>/<root>_<grating>-<filter>_<pid>_<srcid>.spec.fits, e.g. extractions/capers-cos01-v4/capers-cos01-v4_prism-clear_6368_46271.spec.fits. Sampled COSMOS-area root "capers-cos01-v4" (CAPERS survey, JWST program 6368, PI Dickinson — confirmed via the downloaded FITS header: TARGPROP=COSMOS_cat, TARG_RA=150.130, TARG_DEC=2.329, OBSFOLDR=COSMOS_P1). curl -sI on that sample spec.fits returned HTTP/1.1 200 OK, Content-Length 673920, Content-Type binary/octet-stream — confirms it downloads.

4) CORS: NEITHER endpoint sends CORS headers. Tested exactly as specified: `curl -s -o /dev/null -D - -H 'Origin: http://localhost:8000' 'https://s3.amazonaws.com/msaexp-nirspec/extractions/dja_msaexp_emission_lines_v4.4.csv.gz' -r 0-100` returns HTTP/1.1 206 Partial Content with no Access-Control-Allow-Origin (or any access-control-*) header anywhere in the response — grepped full header dump, zero matches. Also confirmed bucket has no public CORS config: `?cors` sub-resource request returns AccessDenied. Also tested grizli-cutout.herokuapp.com/thumb with an Origin header on a GET — likewise no Access-Control-Allow-Origin header returned. OPTIONS preflight against S3 also returns 403. Conclusion: a browser-side fetch()/XHR from a web app on a different origin will be blocked by CORS for both the S3 bucket and the Heroku cutout API; only same-origin proxying, <img>/<a> tag navigation (not subject to CORS), or a server-side backend proxy will work for programmatic (fetch/XHR/JSON parsing) access.

5) grizli-cutout.herokuapp.com API (from live root-page self-doc at https://grizli-cutout.herokuapp.com/ and the fuller https://dawn-cph.github.io/dja/general/api_summary/): endpoints are overlap, thumb, mosfire, cosmos-psf (root page) plus assoc, exposures, assoc_mosaic, nirspec_slits, nirspec_extractions, nirspec_file_redshift, nirspec_trace_overlap (api_summary page). /thumb params: ra, dec (or coords="ra,dec" / sexagesimal "HH:MM:SS.S DD:MM:SS.S" or comma-delimited), size (arcsec, default 4), output (png [default]/fits/fits_weight — YES it returns multi-extension FITS cutouts, not just PNG), filters (comma-separated; RGB auto-chosen via grizli.pipeline.auto_script.field_rgb, or single filter for greyscale), default_filters (hst/jwst), scl (brightness scale), rgb_scl (relative R/G/B scaling), rgb_min, asinh=True (asinh vs Lupton RGB scaling), pl=<n> (power-law wavelength**pl scaling), all_filters=True (grid of per-filter greyscale cutouts), pad (padding between multi-filter tiles), invert, dpi_scale, plus NIRSpec-overlay-specific params nirspec (True/False, overlays NIRSpec slit footprints on the cutout), metafile (e.g. "jw04233001002", restricts overlay to a specific MSA metafile/program-obs), nrs_source, nrs_other, msa_other, nrs_lw, nrs_alpha, and slit="{ra},{dec},{length_arcsec},{width_arcsec},{PA}" (draws an arbitrary custom slit box). Live-tested: `thumb?ra=150.13&dec=2.328&size=15&output=png&scl=2&asinh=True&filters=f444w-clear,f277w-clear,f150w-clear` returned HTTP 200, Content-Type image/png (~628KB RGB PNG). No auth required; served via Heroku (Server: Heroku header), no CORS headers set (see #4).

6) *.spec.fits structure (sample downloaded to /tmp/claude-1005/-otdata2-themiya-minerva/fcc0bdc0-01de-4a5b-92a4-a69c924bb4c7/scratchpad/dja-probe/sample.spec.fits from extractions/capers-cos01-v4/capers-cos01-v4_prism-clear_6368_46271.spec.fits, inspected with /otdata2/themiya/grizli_rebels/bin/python3 + astropy.io.fits): 8 HDUs — PRIMARY (empty, no header keywords beyond boilerplate), SPEC1D (BinTableHDU, 473 rows x 9 cols: wave [um], flux [uJy], err [uJy], sky [uJy], path_corr, npix, flux_sum, profile_sum, var_sum — this is the 1D extracted spectrum table), SCI (ImageHDU, 473x31 float64, BUNIT=microJansky — 2D rectified spectrum), WHT (ImageHDU, 473x31, inverse-variance weight map), PROFILE (ImageHDU, 473x31, spatial profile model), PROF1D (BinTableHDU, 31 rows x 3 cols, 1D spatial profile), BACKGROUND (ImageHDU, 473x31, background model), SLITS (BinTableHDU, 6 rows x 49 cols, metadata about each contributing exposure/slit: filenames, slit geometry, dither/quadrant/row/column IDs etc). The SCI header carries full JWST pipeline provenance (SDP_VER, CAL_VER, OBS_ID, PROGRAM=06368, TARGPROP=COSMOS_cat, TARG_RA/TARG_DEC, INSTRUME=NIRSPEC, DETECTOR, FILTER=CLEAR, GRATING=PRISM, MSAMETFL, etc). Redshift/grade/line-flux info is NOT embedded in the spec.fits itself — it lives only in the separate catalog CSV / nirspec_extractions API (joined by root+file or srcid).

# RISKS
- S3 bucket listing (ListBucket) is denied for anonymous users, so the 'top-level prefixes under extractions/' and 'root directory count' were NOT obtained via authoritative directory enumeration -- they were reconstructed by parsing the v4.5 catalog CSV, which only lists roots that have at least one row in that catalog. There could be additional root directories in the bucket (e.g. proprietary/embargoed data, or non-PRISM-only fields not yet merged into the public CSV) not captured by this method.
- v4.5's catalog/HTML files are only confirmed to exist by guessing typical version-number filenames (v4.4, v4.5, v4.6...v5.2, nirspec_public_v4.5.html, nirspec_graded_v4.html) and probing each with HEAD; a truly newer version with an unexpected filename convention (e.g. a non-sequential jump, dated filename, or renamed catalog) would not have been found this way.
- No official blog post or documentation page was found describing the v4.5 release specifically -- its existence, contents, and exact relationship to v4.4 (e.g. what changed, why no Zenodo archive yet) is inferred only from the file being live on S3 with a later Last-Modified date and containing more rows (89,303 vs presumably fewer in v4.4); did not diff the two catalogs' contents.
- WebFetch tool output is an AI-summarized paraphrase of fetched pages, not verbatim HTML -- exact wording/params from dawn-cph.github.io pages (api_summary, mapview, blog post) should be treated as reliable in substance but summarized in phrasing; recommend a follow-up direct curl+grep on those pages if verbatim text is needed.
- The nirspec_extractions live API's 'output' parameter behavior (json vs csv) wasn't fully nailed down -- requested output=json but got CSV back, suggesting either the default output format ignored the query param or CSV is the fallback; this should be verified before an app depends on JSON output specifically.
- CORS testing only covered GET (with Origin header) and OPTIONS preflight on two specific object URLs / one thumb URL; did not exhaustively test every endpoint (e.g. nirspec_extractions, assoc) for CORS, though the pattern (Heroku app, S3) strongly suggests none of them set CORS headers.
- Did not verify whether grizli-cutout.herokuapp.com's /thumb 'nirspec'/'metafile' overlay params actually work end-to-end (only confirmed they're documented on the api_summary page); did not render an example with nirspec=True to visually confirm overlay behavior.
- Root-count of 219 and the '-v4' suffix pattern reflect the v4.5 catalog snapshot as of 2026-07-18; this is a moving target as DJA periodically adds new programs/roots.
